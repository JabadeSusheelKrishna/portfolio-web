
## 📝 Project Summary: Tamil Nadu Temple Detection AI

### 🏛️ Project Overview

* 
**Goal**: Developed an AI-driven, end-to-end computer vision solution to accurately identify and classify **15 historically significant temples** from Tamil Nadu using user-uploaded images.


* 
**Purpose**: Acts as a virtual guide for tourists and historians by instantly retrieving historical data, visiting hours, and Google Maps navigation upon classification.


* 
**Tech Stack**: Built using **Python, PyTorch, Transformers (Hugging Face), Docker, and Streamlit**.



---

### 🔬 Methodology & Iterative Approaches

To find the best accuracy, the project systematically tested **5 different prediction methodologies**:

* 
**V1 (Baseline Zero-Shot)**: Passed raw temple names directly to OpenAI's CLIP vision-language model, serving as the benchmark.


* 
**V2 (Prompt Engineering)**: Enriched text prompts semantically (e.g., *"A majestic photo of the [Temple Name]..."*) to improve visual-conceptual alignment.


* 
**V3 (Region of Interest)**: Automatically cropped the central **70%** of images to eliminate background noise like clouds and crowds.


* 
**V4 (Image Enhancement)**: Applied an OpenCV sharpening filter to make intricate architectural carvings more prominent for the model encoder.


* 
**V5 (Hybrid FFNN Head)**: Froze the CLIP image encoder and trained a custom, lightweight Multi-Layer Perceptron (MLP) head with ReLU activation and Dropout regularization.

## 📝 Project Summary: Distributed Messaging Queue using RabbitMQ

### 🏗️ Project Architecture & Overview

* **Goal**: Engineered a highly available, fault-tolerant distributed task processing system capable of handling asynchronous, high-concurrency workloads.
* **System Design**: Designed a 4-tier architecture comprising a Client CLI, an Application Layer with multiple Producer nodes, a Messaging Layer, and a Worker Layer.
* **Tech Stack**: Built entirely using **Go (Golang), RabbitMQ (Quorum Queues), gRPC (HTTP/2), Protocol Buffers, AMQP 0-9-1, Docker, and Docker Compose**.

---

### 🧩 Core Component Implementation

* **Client-Side Load Balancing**: Implemented an external client CLI that randomly distributes incoming tasks across available producer nodes via high-performance **gRPC** connections.
* **Intelligent Routing Gateway**: Developed Producer nodes that act as gateways, converting incoming gRPC requests into **AMQP 0-9-1** messages and executing custom routing logic (Direct routing for individual queues and Fan-out routing for simultaneous multi-queue task processing).
* **High Availability Broker Cluster**: Deployed a **4-node RabbitMQ Cluster** utilizing shared Erlang Cookies for node authentication and auto-discovery. Enabled data resilience by utilizing **Raft-consensus-based Quorum Queues**, message persistence, and durable queues to eliminate single points of failure.
* **Reliable Worker Pool**: Built Consumer workers utilizing **Manual Acknowledgments** to prevent data loss in the event of unexpected runtime crashes. Configured Quality of Service (**QoS Prefetch = 1**) to dynamically load-balance tasks and prevent consumer overloading.

---

### 📊 Simulation Performance & Key Metrics

Evaluated system limits under a high-concurrency load generation test to analyze throughput and uncover operational bottlenecks:

* **High-Volume Burst Test**: Submitted a heavy burst of **60 tasks within a 6-second window** (sustained rate of **10 tasks/second**) across Direct, Fanout, Topic, and Header exchanges.
* **100% Reliability Rate**: Successfully processed and completed **60 out of 60 tasks**, maintaining a perfect **100% routing accuracy** and zero message loss under intense load.
* **System Throughput**: Maintained a processing throughput of **7.70 tasks/second** across the cluster.
* **Bottleneck Identification**: Logged an average processing time of **1.95s** per task, resulting in an average queue latency of **33.60s** and a P95 latency of **64.00s**. Identified that a 2-consumer pool creates a sequential processing bottleneck during micro-bursts, concluding that adding more worker instances will scale down wait times linearly.

---

### 🌟 Future Enhancements & Production Roadmap

* **Full-Stack Observability**: Integrating **Prometheus metrics**, **Grafana dashboards**, and **OpenTelemetry** for end-to-end distributed tracing.
* **Advanced Error Handling**: Implementing **Dead Letter Queues (DLQ)** alongside exponential backoff retry policies to capture and manage failed tasks.

## 📝 Project Summary: Provably Secure End-to-End Cryptographic & MPC Stack

### 🛠️ Core Engineering Blueprint & Design Constraints

* 
**The No-Library Constraint**: Engineered a massive, production-grade cryptographic dependency chain completely from scratch using **Python and React**. Substitutions of standard cryptographic libraries (e.g., OpenSSL, PyCryptodome) were forbidden.


* 
**Strict Lineage Rule**: Implemented low-level operations dynamically so that high-level applications logically trigger custom underlying primitives down to arbitrary-precision arithmetic and OS randomness.


* 
**Interactive React Explorer Component**: Developed an incremental React/Vite-based web application calling core backend functions compiled to WebAssembly/Local APIs. Features live data-flow rendering, a step-through DAG routing table, and an empirical security distinguishing game emulator.



---

### 🧩 Core Mathematical Modules & Security Implementations

#### 📦 Phase 1: Symmetric Cryptography & The Minicrypt Clique (PA #1 – #6)

* 
**Foundation Bootstrap**: Built a length-doubling Pseudorandom Generator (PRG) via iterative Goldreich-Levin hard-core bit extraction over a custom Discrete Logarithm Problem (DLP) group.


* 
**Functional Primitives**: Constructed a deep binary **GGM Tree Pseudorandom Function (PRF)** mapping arbitrary bit sequences, verified via NIST SP 800-22 statistical test suites.


* 
**Cipher Block Operations**: Formulated a uniform API handling multi-block data streams via **CBC, OFB, and randomized Counter (CTR) modes** alongside custom PKCS padding layouts.


* 
**Malleability Hardening**: Designed variable-length fixed-tag MAC systems to scale up a malleable CPA-secure cipher into an indistinguishable, **CCA2-secure Active-Defense Encrypt-then-MAC engine**.



#### 🔑 Phase 2: Hashing, Data Integrity, & Public Key Cryptography (PA #7 – #17)

* 
**Resilient Hash Framework**: Created a generic **Merkle-Damgård Transform** engine featuring custom block-strengthening pads. Proved security metrics by deploying Floyd's Tortoise-and-Hare cycle detection to run automated **Birthday Boundary Attacks**.


* 
**HMAC Double-Hash Structure**: Built a complete secure **HMAC** system over custom DLP-based compression blocks , preventing classic length-extension state forgery.


* 
**Asymmetric Infrastructure**: Coded a custom **Miller-Rabin Probabilistic Primality Tester** ($k=40$ rounds, error $\le 4^{-40}$) to generate 1024/2048-bit safe primes ($p=2q+1$).


* 
**Cryptomania Cryptosystems**: Created textbook **RSA** and modular **ElGamal PKC engines**. Optimized honest-party operations using **Garner's CRT Recombination algorithm** for a **3–4x decryption speedup**. Implemented adaptive padding rules to neutralize **Håstad's Broadcast and Bleichenbacher oracle attacks**.



#### 🤝 Phase 3: Secure Multi-Party Computation / MPC Land (PA #18 – #20)

* 
**Oblivious Transfer Core**: Engineered an atomic **1-out-of-2 Bellare-Micali Oblivious Transfer protocol** leveraging trapdoor elimination to guarantee perfect computational receiver privacy.


* 
**Complete Gate Synthesis**: Developed zero-communication **Secure XOR** and **NOT** layers via local additive secret sharing over $\mathbb{Z}_2$, alongside a protocol-driven **Secure AND Gate** subroutine.


* 
**Yao's Functional Completeness Execution**: Rolled out an MPC compiler that represents arbitrary 2-party polynomial functions as directed acyclic graphs (DAGs). Successfully evaluated the **Millionaire’s Problem (unbounded $x > y$)**, bit-wise equality arrays, and modular ripple-carry additions with a zero-knowledge, fully simulatable transaction transcript.

## 📝 Project Summary: Agent-Based Simulation of Commuter EVs

### 🌐 Project Overview & Core Architecture

* **Goal**: Engineered an agent-based simulation system to model and analyze the daily movement, routing, and charging behavior of commuter electric vehicles (EVs) across a synthetic city network.
* **System Design**: Designed an interactive environment where discrete vehicle agents dynamically navigate a road network connecting residential (home), commercial (office), retail (shops), and hospitality (restaurants) nodes.
* **Data Pipelines**: Built a discrete-time simulator running on 1-minute intervals that concurrently updates, captures, and logs the real-time operational state of all vehicle agents and infrastructure components.

---

### ⚙️ Core Technical Implementations & Features

* **Intelligent Routing & Infrastructure Allocation**:
* Implemented and benchmarked **three distinct allocation algorithms** to dynamically route vehicles with low battery parameters to optimal charging infrastructure.
* **Nearest Station Allocation**: Leveraged **Dijkstra’s algorithm** to compute the absolute shortest path to the geographically closest available charging hub.
* **Least Queue Allocation**: Routed agents to hubs with the smallest active waiting queue to systematically minimize vehicle wait times.
* **Random Station Allocation**: Handled baseline stochastic load-balancing by distributing agents randomly across open queues.


* **Dynamic Physics & Consumption Model**:
* Modeled the city transit infrastructure as a topological graph of nodes and edges.
* Programmed vehicles to read edge-level speed limits dynamically, adjusting their instant velocity, step-wise distance covered, and battery discharge curves accordingly.


* **Stochastic Behavioral Scheduling Engine**:
* Assigned each agent a unique weekly transit profile mapping specific shopping and dining destinations.
* Modeled finite-state machine transitions (e.g., *Moving to Shop, At Location*) executed by a temporal scheduler handling randomized, agent-consistent visit constraints across a 7-day calendar matrix.



---

### 📊 Simulation Processing & Output Telemetry

* **Ingestion and Setup**: Initialized simulation space by parsing spatial city layouts and infrastructure positions directly from input **GeoJSON files**.
* **Granular State Logging**: Captured micro-level data points every minute—including exact coordinates, instant speed values, remaining battery percentages, grid queue lengths, and active power draw counts.
* **Comparative Analytics Data Store**: Exported granular execution outputs into distinct, structurally indexed **JSON files** mapped to each allocation method, generating a clean data source for visual optimization analysis.

---

* **Multi-Algorithmic Testing Framework**: Successfully built and automated the isolated execution of **3 structural optimization policies** to analyze queue delays and route bottlenecks.
* **High-Fidelity Telemetry Logging**: Developed a 1-minute interval data-logging engine recording vehicle parameters and power grid statuses to a structured JSON database.
